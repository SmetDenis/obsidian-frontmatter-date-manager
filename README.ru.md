Автоматически обновляет даты `created`, `updated` и `viewed` в YAML frontmatter при редактировании заметок в Obsidian.

## Зачем нужен этот плагин?

- **Поддерживать даты вручную утомительно.** Каждый раз вручную править `created` и `updated` во frontmatter - значит ошибаться и сбивать рабочий ритм.
- **В Obsidian нет встроенного управления датами во frontmatter.** Он отслеживает `ctime`/`mtime` на уровне файловой системы, но не записывает и не поддерживает свойства с датами внутри заметок автоматически.
- **Инструменты синхронизации вызывают ложные обновления.** Obsidian Sync, iCloud, Syncthing, Dropbox и синхронизация на основе Git изменяют файлы без реальных изменений содержимого. Без хеширования содержимого каждая синхронизация запускала бы обновление дат - порождая шум и потенциально бесконечные циклы синхронизации.
- **Шаблоны и плагины автоматизации конфликтуют.** Templater, Daily Notes, QuickAdd и подобные плагины создают и тут же изменяют файлы. Без настраиваемой задержки даты записывались бы до того, как шаблон полностью применён, и оказывались бы неверными.
- **В существующих хранилищах нет дат.** Когда вы подключаете плагин к хранилищу с сотнями или тысячами заметок, нужен способ массово заполнить даты из дат файловой системы - а не обновлять каждую заметку по одной.
- **Ручной ввод приводит к разнобою форматов.** В разных заметках оказываются `2024-01-15`, `Jan 15, 2024`, `15.01.2024` и другие варианты. Плагин обеспечивает единый настраиваемый формат во всём хранилище.
- **Нет автоматического отслеживания «последнего открытия».** Obsidian знает, когда файл был изменён, но не имеет представления о том, когда вы в последний раз его *читали*, - и ни один другой плагин не записывает это во frontmatter. Этот плагин может опционально проставлять дату `viewed` при каждом открытии заметки, делая её доступной для запросов через Dataview для интервального повторения, рабочих процессов ревью и дашбордов в духе «что я не открывал уже несколько месяцев?».

## Возможности

- Автоматическое обновление поля `updated` при изменении файла (синхронизируется с `mtime`)
- Автоматическая установка поля `created` для новых файлов (синхронизируется с `ctime`)
- Автоматическая установка поля `viewed` при открытии файла - уникальная возможность, которой нет в других плагинах (отключена по умолчанию)
- Подсчёт того, как часто вы редактируете каждую заметку (`updated_count`, отключён по умолчанию) - приблизительный сигнал активности, который можно сортировать или фильтровать в Bases/Dataview, чтобы найти самые редактируемые заметки
- Настраиваемый формат даты (использует синтаксис [date-fns](https://date-fns.org/v4.1.0/docs/format))
- Поддержка часовых поясов с автодополнением часовых поясов IANA
- Свойства типа «строка» и «число» (число удобно для меток времени Unix)
- Правила фильтрации файлов в стиле gitignore с предпросмотром и валидацией
- Настраиваемый минимальный интервал между обновлениями
- Задержка для только что созданных файлов (совместимость с Templater, Daily Notes и др.)
- Хеширование содержимого SHA-256 для обнаружения реальных изменений (предотвращает ложные обновления от инструментов синхронизации)
- Режим обнаружения изменений: только тело заметки, только свойства или и то и другое
- Исключение свойств из обнаружения изменений
- Запуск команды после обновления дат
- Массовое заполнение дат из собственных дат каждого файла на диске, с предпросмотром в режиме пробного прогона
- Переименование свойства во всех заметках (миграция старых имён с предпросмотром)
- Переформатирование существующих дат из одного формата в другой (разбор старого, запись нового, с предпросмотром)
- Каждый массовый предпросмотр разбит на страницы (Назад/Далее), показывает все затронутые файлы (без ограничения числа строк) и может выгрузить полный список изменений в файл TSV на компьютере (сохраняется в системную папку загрузок, никогда не записывается в ваше хранилище)
- Переключение автообновления через палитру команд или строку состояния
- Приостановка автообновления на 5 минут с автоматическим возобновлением
- Многоязычный интерфейс, следующий за языком приложения Obsidian - английский и русский проверены вручную, плюс 19 базовых переводов, все с попадейным откатом на английский
- Работает на компьютере и на мобильных устройствах

## Установка

### Сообщество плагинов

В Obsidian откройте «Настройки» > «Плагины сообщества» > «Обзор», найдите **Frontmatter Date Manager** и нажмите «Установить».

### Установка вручную

Скачайте `main.js`, `manifest.json` и `styles.css` из
[последнего выпуска](https://github.com/SmetDenis/obsidian-frontmatter-date-manager/releases/latest)
в `<vault>/.obsidian/plugins/frontmatter-date-manager/`.

## Использование

Плагин работает автоматически после установки. Когда вы редактируете markdown-файл, он обновляет свойство `updated` текущим временем изменения. Если свойство `created` отсутствует, он устанавливает его равным времени создания файла. При желании включите дату `viewed` в настройках, чтобы фиксировать, когда вы в последний раз открывали каждую заметку.

Настройте поведение в разделе **«Настройки» -> Frontmatter Date Manager**.

### Команды

| Команда                                | Описание                                             |
|----------------------------------------|---------------------------------------------------------|
| **Update timestamps for current file** | Вручную запустить обновление дат для активной заметки |
| **Toggle auto-update on/off**          | Включить или отключить автоматическое обновление дат           |
| **Pause auto-update for 5 minutes**    | Временно приостановить обновления с автоматическим возобновлением         |

**Индикатор в строке состояния** - показывает текущее состояние (`Paused` или `Paused (Xm)`); щёлкните, чтобы включить или отключить автообновление.

### Дата изменения раньше даты создания

| Настройка                          | По умолчанию    | Описание                                                                                                                       |
|----------------------------------|------------|----------------------------------------------------------------------------------------------------------------------------------|
| `How to fix out-of-order dates`  | `disabled` | Что делать, когда дата последнего изменения раньше даты создания. Применяется к автоматическим правкам; `disabled` означает только обнаружение. |
| `Ignore tiny differences (seconds)` | `0`     | Игнорировать нарушенный порядок дат, когда разрыв меньше этого значения. Полезно, чтобы подавить субсекундный сдвиг часов.                            |

Доступные стратегии: `Set creation date to the last-edited date`, `Set last-edited date to the creation date`, `Set both to the most recent date`.

## Примеры форматов даты

| Строка формата           | Пример вывода            |
|-------------------------|---------------------------|
| `yyyy-MM-dd'T'HH:mm:ss` | 2026-04-12T14:30:00       |
| `yyyy-MM-dd HH:mm:ss`   | 2026-04-12 14:30:00       |
| `dd.MM.yyyy HH:mm`      | 12.04.2026 14:30          |
| `t`                     | 1776268200 (секунды Unix) |
| `T`                     | 1776268200000 (миллисекунды Unix)   |

> **Примечание.** Этот плагин использует **date-fns**, а не Moment.js. Частая миграция: `YYYY` -> `yyyy`, `DD` -> `dd`.

## Поиск самых редактируемых заметок (необязательно)

Когда включено **«Считать правки»**, свойство `updated_count` - это обычное число, которое можно сортировать и фильтровать как угодно. Плагин записывает только число - построение представления остаётся за вами. Например, запрос Dataview для заметок, которые вы редактируете чаще всего:

````markdown
```dataview
TABLE updated_count, updated
WHERE updated_count
SORT updated_count DESC
LIMIT 20
```
````

Или в **Bases** в Obsidian отсортируйте по `updated_count` по убыванию (при желании отфильтруйте по недавнему `updated`). Счётчик - это приблизительный сигнал активности, а не точная история: он начинается с момента, когда вы включаете эту возможность. Он увеличивается один раз за *сеанс* редактирования, а не за нажатие клавиши: быстрые правки в пределах окна **«Минимум секунд между обновлениями»** считаются как одна. Запуск команды **Update timestamps for current file** тоже учитывается; массовые операции (заполнение/переформатирование/переименование) переписывают даты, не меняя счётчик.

## FAQ

### Первая установка

> Изменит ли плагин все мои существующие заметки, когда я впервые его включу?

Нет. Плагин обрабатывает файл только тогда, когда вы его редактируете. При первой загрузке он в фоне строит хеш-кэш существующих файлов, чтобы подготовиться к обнаружению изменений, но в этом процессе никогда не записывает даты. Ваше хранилище остаётся нетронутым, пока вы действительно не отредактируете заметку.

> Как добавить даты в заметки, написанные до установки?

Используйте «Настройки» → «Массовые операции» → «Задать даты из собственных дат файла». Плагин читает даты создания и изменения каждого файла на диске и записывает их в свойства вашей заметки, с предпросмотром в режиме пробного прогона, чтобы вы могли всё проверить перед применением. Режим по умолчанию - «Только заполнить отсутствующие»: существующие даты не перезаписываются. Если ваше хранилище синхронизируется через iCloud или Obsidian Sync, эти даты на диске могли быть сброшены службой синхронизации - внимательно изучите предпросмотр.

> Я использую Templater / Daily Notes / QuickAdd. Будет ли плагин с ними конфликтовать?

Нет. Плагин ждёт 5 секунд (настраивается: «Настройки» → «Поведение» → «Дополнительно» → «Задержка для новых файлов») перед обработкой только что созданных файлов, давая плагинам шаблонов время завершить работу.

> Нужно ли сначала вручную добавлять свойства в каждую заметку?

Нет. Если у заметки ещё нет свойств, плагин создаёт блок `---` и вставляет даты при следующем редактировании. Если свойства уже есть, он добавляет свойства с датами рядом с существующими.

> Какой формат даты лучше всего подходит для Dataview?

Формат по умолчанию `yyyy-MM-dd'T'HH:mm:ss` (ISO 8601) работает из коробки. Dataview может разбирать, сортировать и сравнивать его нативно.

> Плагин использует date-fns, а не Moment.js. Это меня касается?

Только если вы настраиваете формат даты. Ключевое отличие: используйте `yyyy` (а не `YYYY`) для года, `dd` (а не `DD`) для дня. Плагин показывает подсказку в настройках, если обнаруживает формат в стиле Moment.js.

### Повседневное использование

> Я включил даты «viewed», но в некоторых заметках они не появляются.

Дата открытия записывается только при открытии файла. Заметки, которые вы не открывали с момента включения этой возможности, ещё не будут иметь это поле. К записи `viewed` применяются те же правила фильтрации и настройка минимального интервала.

> Я отредактировал теги или псевдонимы, но `updated` не изменился. Это баг?

Нет. По умолчанию обнаружение изменений смотрит только на тело заметки - дату обновляют только изменения ниже блока свойств. Чтобы учитывать изменения свойств, переключите «Настройки» → «Обнаружение изменений» → «Что считается изменением» на «Both».

> Вызовет ли синхронизация (iCloud / Obsidian Sync / Dropbox) ложные даты?

Нет. Плагин сравнивает содержимое файла с помощью хеширования SHA-256. Если служба синхронизации перезаписывает файл, не меняя его содержимое, хеш совпадает, и дата не обновляется. Включено по умолчанию.

> Я переименовал или переместил заметку. Потеряет ли плагин её из виду?

Нет. Запись в хеш-кэше автоматически переносится на новый путь. Существующие даты сохраняются.

> Я изменил формат даты. Будут ли старые даты сконвертированы?

Не автоматически. Используйте «Настройки» → «Массовые операции» → «Переформатировать даты», чтобы привести все значения к единому виду. Плагин автоматически распознаёт существующие форматы (ISO 8601, европейский, американский, числовые метки времени) и переписывает их в вашем текущем формате. Перед применением просмотрите все изменения.

> Дата вроде `01/05/2024` может означать 5 января или 1 мая. Что произойдёт?

Такие неоднозначные даты вида день/месяц по умолчанию остаются без изменений - плагин никогда не угадывает. В предпросмотре показано, сколько таких найдено, и предлагается выбор в один клик (сначала день или сначала месяц), заранее подсказанный по вашему системному региону, так что вы сами решаете до того, как что-то будет переписано. Даты с единственным допустимым прочтением (например, `25/12/2024`) конвертируются всегда.

> Я переименовал свойство (например, `created` → `date_created`). Что с существующими файлами?

Используйте «Настройки» → «Массовые операции» → «Переименовать свойство». Введите старое и новое имена свойства, просмотрите затронутые заметки, затем примените. Вы можете выбрать, удалять ли старое свойство или оставить оба.

> Я изменил часовой пояс. Будут ли старые даты пересчитаны?

Нет. Тот же принцип - старые значения остаются нетронутыми. Новые записи используют новый часовой пояс.

> Что будет, если у заметки сломанный YAML frontmatter?

Плагин пропускает такой файл и показывает уведомление с путём к файлу и подробностями ошибки. Он никогда не записывает в файл с некорректным YAML. Исправьте синтаксис, и плагин подхватит файл при следующем редактировании.

> Я часто сохраняю. Будет ли дата обновляться при каждом сохранении?

Нет. Есть минимальный интервал в 30 секунд между обновлениями (настраивается: 5-300 секунд) плюс задержка (debounce) в 2 секунды, так что быстрые правки объединяются в одну запись даты.

## Синхронизация и контроль версий

Плагин хранит локальный файл кэша `hash-cache.json` внутри своего каталога данных (`.obsidian/plugins/frontmatter-date-manager/`). Этот файл содержит хеши SHA-256, используемые для обнаружения изменений содержимого. Он перестраивается автоматически при запуске, поэтому исключать его безопасно и рекомендуется.

**Почему стоит исключить:** кэш обновляется при каждом редактировании файла, поэтому несколько устройств меняют его независимо - что вызывает частые конфликты синхронизации и лишний трафик. Поскольку он перестраивается автоматически, синхронизация не даёт никакой пользы.

Добавьте в ваш `.gitignore`:

```
.obsidian/plugins/frontmatter-date-manager/hash-cache.json
```

Для **Obsidian Sync**: файл уже исключён автоматически (Sync не синхронизирует файлы данных плагинов, кроме `data.json`).

Для **iCloud, Syncthing, Dropbox или другой файловой синхронизации**: добавьте `hash-cache.json` в список игнорирования/исключений вашего инструмента синхронизации для каталога плагина.

## Конфиденциальность и возможности

Этот плагин полностью локальный. У него нет бэкенда, он не делает сетевых запросов и не собирает никакой телеметрии или аналитики. В карточке оценки плагина сообщества перечислены возможности, которые может использовать код плагина; вот для чего именно этот плагин использует каждую из них:

- **Читает markdown-файлы в вашем хранилище.** Массовые инструменты (заполнить даты, переименовать свойство, переформатировать даты, найти даты в нарушенном порядке, перестроить кэш обнаружения изменений) работают по всему хранилищу, поэтому перечисляют ваши markdown-заметки через `getMarkdownFiles()` в Obsidian. Плагин никогда не перечисляет не-markdown-файлы (`getFiles()` не используется), так что вложения, изображения и другие бинарные файлы никогда не затрагиваются.
- **Записывает только настроенные свойства с датами.** Все изменения проходят через `processFrontMatter()` в Obsidian, который затрагивает только настроенные вами свойства `created` / `updated` / `viewed` и оставляет тело заметки, порядок ключей, комментарии и не относящиеся к делу свойства нетронутыми.
- **Записывает один вспомогательный файл в собственной папке плагина.** Кэш обнаружения изменений SHA-256 (`hash-cache.json`) записывается внутри `.obsidian/plugins/frontmatter-date-manager/`, никогда в ваши заметки.
- **Только локальный экспорт.** Кнопка «Скачать полный предпросмотр» сохраняет список изменений как локальный файл `.tsv` через браузер и не записывает никакого файла в ваше хранилище. Скачивание файла доступно только на компьютере - на мобильных устройствах полный список изменений остаётся доступным для чтения в таблице на экране.

## Языки

Интерфейс плагина автоматически следует за настройкой языка самого Obsidian - отдельного параметра языка задавать не нужно. В нём есть переводы на 21 язык (арабский, немецкий, английский, испанский, персидский, французский, индонезийский, итальянский, японский, корейский, нидерландский, польский, португальский, бразильский португальский, русский, тайский, турецкий, украинский, вьетнамский, а также китайский упрощённый/традиционный), и любой текст, который перевод не покрыл, откатывается на английский, так что интерфейс никогда не остаётся пустым. Английский и русский проверены вручную; остальные - базовые переводы, и **улучшения очень приветствуются** - см. раздел «Translations» в [`CONTRIBUTING.md`](CONTRIBUTING.md).

## Разработка

```bash
make              # Show all available commands
make install      # Install dependencies
make pre-commit   # Run all checks (format, lint, test, build)
make local-test   # Build and copy plugin to local vault
```

Чтобы использовать `make local-test`, задайте `OBSIDIAN_VAULT_TEST` в окружении вашей оболочки или передайте его напрямую: `make local-test OBSIDIAN_VAULT_TEST=/path/to/vault`.
